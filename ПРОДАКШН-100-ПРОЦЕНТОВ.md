# Сайт на домене «на 100%»: что уже автоматизировано и что только вручную

Ниже честно: **никакой код не может за вас** подписать договор с банком, завести DNS у регистратора или выдать ключ в личном кабинете СДЭК. Остальное в этом проекте **уже заведено на автоматику** (GitHub Actions + скрипты на VPS).

---

## 1. Что уже автоматизировано в репозитории (вам не писать скрипты)

| Автоматика | Что делает |
|------------|------------|
| **`.github/workflows/deploy-reg-vps.yml`** | По push в `main` (изменения в `server/`, lockfile и т.д.) заходит по SSH на VPS, `git fetch` + `reset --hard origin/main`, патчит `.env` из **GitHub Secrets**, `npm ci`, **pm2 restart** API, проверка `/api/health`, при заданных секретах — **`install-api-proxy.sh`** (nginx + HTTPS для поддомена API). |
| **`.github/workflows/deploy-reg-ftp.yml`** | Собирает фронт с **`VITE_API_BASE_URL` из Secrets** и заливает **`dist/`** на хостинг по FTP. |
| **`server/deploy/install-api-proxy.sh`** | Поднимает прокси **HTTPS → Node** на хосте из секрета **`API_PUBLIC_HOST`** (например `api.ваш-сайт.ru`). |
| **`ecosystem.config.cjs`** | PM2 запускает API в **production** на порту **8787**. |
| **`npm run prod:check`** | Проверяет `.env`: обязательные поля + **СДЭК / Точка / Telegram / почта / БД** (подсказки, чего не хватает). |
| **`npm run zip:dist`** | Сборка + архив для ручной загрузки фронта, если без GitHub. |
| **`npm run go`** | Локально: API + Vite одной командой. |
| **`.github/workflows/weekly-database-email-backup.yml`** | По cron (и вручную): SSH на VPS, скрипт **`server/scripts/weekly-email-backup.mjs`**. Расписание срабатывает, если в GitHub **Variables** задано **`WEEKLY_EMAIL_BACKUP_ENABLED`** = `1` (или запуск вручную из Actions). |

**Фронт и бэк деплоятся разными workflow:** на VPS по SSH уходит только код **`server/`** (и зависимости репозитория на сервере) — см. **`deploy-reg-vps.yml`**. Сборка фронта и FTP — отдельно **`deploy-reg-ftp.yml`** (всегда выгружается **`dist/`** после `npm run build`). Не путайте с ручной заливкой корневого `index.html` с `/src/main.tsx` — от этого «белый экран» (нужен именно **`dist/index.html`**).

На сайте есть маршрут **`/debug`** (вкладки Telegram и СДЭК): удобно проверить OAuth СДЭК и доставку без полного оформления заказа.

В деплой бэкенда добавлено прокидывание **`JWT_SECRET`** и **`DATABASE_URL`** из GitHub Secrets в `.env` на сервере (раньше их не патчило).

## 2. Что **физически нельзя** автоматизировать (только вы или ваш банк/провайдер)

1. **Домен** — куплен/прикреплён у регистратора; **DNS-записи** (A/AAAA на IP VPS, CNAME на хостинг и т.д.) выставляются в панели регистратора или у хостинга.
2. **SSL** — для сценария с VPS скрипт тянет **Let's Encrypt** через certbot (нужен **email** в секрете `CERTBOT_EMAIL` и открытые порты **80/443**).
3. **Точка (эквайринг, API)** — токены и ID выдаются **в кабинете Точки**; их вы копируете в Secrets / `.env`.
4. **СДЭК** — логин/секрет API из **личного кабинета СДЭК** для интеграции.
5. **Telegram** — токен у **@BotFather**, `chat_id` канала/чата; либо настройка **relay** на Edge (если используете).
6. **Почта для бэкапов** — SMTP-пароль (часто «пароль приложения» у Mail.ru и т.п.).
7. **PostgreSQL** (если нужен вместо файла `db.json`) — создание БД у хостинга и строка **`DATABASE_URL`**; права на таблицы (см. `server/sql/pg-bootstrap.sql` и комментарии в коде при ошибке прав).

Без пунктов 3–6 соответствующие кнопки на сайте **просто не к чему подключить** — это не баг, а отсутствие ключей.

## 3. Что **вам руками** сделать один раз (чеклист)

### A. GitHub (репозиторий проекта)

1. Откройте репозиторий → **Settings → Secrets and variables → Actions**.
2. Заполните секреты **минимум для своего сценария** (Reg VPS + FTP, как у вас в workflow):

**Сервер (VPS), SSH-деплой**

- `VPS_HOST`, `VPS_USER`, `VPS_SSH_KEY`, `VPS_PORT` (если не 22), `VPS_APP_PATH` — путь к **клону этого репозитория** на сервере.
- `ALLOWED_ORIGINS` — через запятую, **точные** URL сайта в браузере, например:  
  `https://coffeenechai.ru,https://www.coffeenechai.ru`
- **`JWT_SECRET`** — тот же длинный ключ, что в локальном `.env` (или новый из `npm run env:secret`).
- **`DATABASE_URL`** — если используете Postgres; если нет — не задавайте, данные останутся в `server/data/db.json` на VPS (делайте бэкап диска).
- Точка: `TOCHKA_JWT_TOKEN`, `TOCHKA_CUSTOMER_CODE`, при необходимости `TOCHKA_MERCHANT_ID`, `TOCHKA_TERMINAL_ID`, `TOCHKA_CLIENT_ID`, `TOCHKA_INVOICE_ACCOUNT_ID`.
- СДЭК: `CDEK_ACCOUNT`, `CDEK_SECRET`; при **тестовых** ключах СДЭК — ещё `CDEK_API_URL`=`https://api.edu.cdek.ru/v2` (иначе на бою OAuth: «No such account secure»).
- Telegram: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` **или** `TELEGRAM_RELAY_URL` + `TELEGRAM_RELAY_SECRET`; при блокировке исходящего HTTPS — `TELEGRAM_HTTPS_PROXY` / `HTTPS_PROXY`.
- Почта бэкапа: `BACKUP_EMAIL_TO`, `BACKUP_SMTP_HOST`, `BACKUP_SMTP_USER`, `BACKUP_SMTP_PASS`, при необходимости порт/secure.
- `PUBLIC_API_BASE_URL` — если ссылки на загрузки должны идти с `https://api.…` (см. комментарий в `.env.example`).
- **HTTPS API на отдельном поддомене:** `API_PUBLIC_HOST` (например `api.coffeenechai.ru`), `CERTBOT_EMAIL`.

**Фронт (FTP)**

- **`VITE_API_BASE_URL`** — **обязательно** полный URL до API с суффиксом `/api`, **совпадающий** с тем, что открывается в браузере после nginx, например:  
  `https://api.coffeenechai.ru/api`  
  Если здесь ошибка — сайт «откроется», но заказы и корзина не достучатся до бэкенда.
- `FTP_HOST`, `FTP_USERNAME`, `FTP_PASSWORD`, `FTP_PORT`, `FTP_PROTOCOL`, `FTP_TARGET_DIR`.
- Опционально: `VITE_YANDEX_MAPS_API_KEY` для карт СДЭК.

3. На **VPS один раз** вручную: `git clone` репозитория в `VPS_APP_PATH`, установить **Node**, **pm2**, **python3** (для `patch_dotenv.py`), создать первый `.env` или дать workflow всё патчить через Secrets.

4. **Права sudo без пароля** для пользователя деплоя на команды nginx/certbot — иначе `install-api-proxy.sh` не допишет конфиг (это настройка Linux, не кода).

### B. После настройки Secrets

- Пуш в **`main`** с изменениями в `server/` или во фронте — сами запустятся workflows (или **Run workflow** вручную в GitHub Actions).
- Локально перед пушем желательно:  
  **`npm run prod:check`** — увидите, чего не хватает в `.env` (на сервере то же должно оказаться через Secrets).

---

### C. Админка Telegram на Edge (если пользуетесь)

В Secrets для **сборки** фронта (или в `.env` перед `npm run build`) добавьте при необходимости:

- `VITE_TELEGRAM_EDGE_BASE_URL`
- `VITE_TELEGRAM_EDGE_ANON_KEY`

И пересоберите фронт (push в пути фронта или ручной workflow).

## 4. Как убедиться, что «всё на 100%» по цепочке

1. Откройте **`https://ваш-api-домен/api/health`** — должен быть JSON `ok`.
2. Откройте **сайт** → попробуйте каталог / корзину (запросы идут на `VITE_API_BASE_URL`).
3. **СДЭК** — оформление доставки в тестовом заказе.
4. **Точка** — тестовая оплата по инструкции банка (тестовый контур, если выдан).
5. **Telegram** — оформите тестовую заявку / заказ и проверьте канал.
6. **Бэкап на почту** — кнопка в админке после заполнения `BACKUP_SMTP_*`.

## 5. Кратко: что остаётся **именно вам**

| Действие | Зачем |
|----------|--------|
| Заполнить **GitHub Secrets** (таблица выше) | Чтобы CI сам прописал `.env` на VPS и собрал фронт с верным API URL. |
| Настроить **DNS** у регистратора | Чтобы домен и `api.` указывали на ваш VPS / хостинг. |
| Один раз подготовить **VPS** (git, node, pm2, sudo) | Чтобы SSH-деплой не падал. |
| Взять ключи в **ЛК Точка, СДЭК, Telegram, почты** | Другой способ их получить невозможен. |
| При Postgres — создать БД и **`DATABASE_URL`** | Надёжнее, чем один файл `db.json` на диске. |
| Запускать **`npm run prod:check`** перед важным релизом | Быстрая самопроверка без программиста. |

Если нужен только «сайт-картинка» без оплаты и доставки — достаточно фронта и простого хостинга; **«всё на 100%»** без ключей интеграций в природе не существует.

Для жизни на своём компьютере по-прежнему см. **`НАЧАЛО-БЕЗ-КОДА.md`**.
